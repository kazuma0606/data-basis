# 既存データの問題点

## 現状のDB構成

```
ECシステム（2015年頃に外部ベンダーが構築）
└─ MySQL on オンプレ or VPS
└─ EC-CUBEベースのカスタム改修

POSシステム（2008年導入、メーカー保守）
└─ SQL Server（ベンダーロックイン）
└─ スキーマ変更はベンダー対応が必要・有償

会員アプリ（2021年にアプリ会社が構築）
└─ PostgreSQL（アプリ会社が管理、社内に権限なし）
└─ 電話番号ベースのID体系（ECと全く別）

在庫・基幹システム（2000年代の老舗ERP）
└─ Oracle（保守費が高い）
└─ 商品コード体系がECと微妙に違う
```

## 顧客マスタの問題

### 名寄せされていない

同一人物が3システムに別々に存在している。

```
EC:  user_id=1042  田中 太郎  tanaka@gmail.com   2018年登録
POS: member_id=552 タナカ タロウ  090-xxxx-xxxx  2015年登録
App: uid=a8f3c    田中太郎    tanaka@gmail.com   2022年登録
```

名寄せ（同一人物の紐付け）が一切されていない。

### TTLがなく顧客マスタが腐敗している

```
会員データ 20万件の実態
  ├─ アクティブ（直近1年以内に購買）:  約 4万人
  ├─ 休眠（1〜3年間購買なし）:        約 8万人
  ├─ 事実上チャーン（3年以上なし）:   約 6万人
  └─ 退会処理漏れ・重複・住所不明:    約 2万件
```

「会員20万人」という数字が社内で一人歩きしており、
マーケが20万人ベースで施策を立案しているが実態は4万人しかアクティブでない。

### 消せない理由

- 「消したら何か壊れるかもしれない」という恐怖で誰も手を付けられない
- 削除の判断基準・ルールが存在しない
- 退会処理が手動フローのため、申請があっても処理漏れが発生している

## データ品質のばらつき

| フィールド | 問題 |
|---|---|
| 電話番号 | `090-1234-5678` `09012345678` `+81-90-...` が混在 |
| 生年月日 | POSは和暦（S55）、ECは西暦（1980）|
| 住所 | 都道府県が「東京都」「東京」「13」で混在 |
| 氏名 | カタカナ・ひらがな・漢字が統一されていない |
| メールアドレス | 退会後も残っている、バウンスしているものが多数 |
| 商品コード | ECとPOSで体系が微妙に異なる（マスタ統合されていない） |

## ログの断絶

| ログ種別 | 問題 |
|---|---|
| ECの商品閲覧ログ | 3ヶ月分しかない（コスト削減で定期削除していた） |
| POSの購買履歴 | 10年分あるが古いものはCSVでアーカイブ（参照困難） |
| アプリの行動ログ | アプリ会社のサーバーにあり、エクスポート手順不明 |

## データ基盤で解決すべきこと

```
├─ 名寄せ（Entity Resolution）  ← 3システムの同一顧客を統合
├─ データクレンジング           ← 電話・住所・氏名の正規化
├─ チャーン判定・TTL管理        ← アクティブ/休眠/チャーンの分類
├─ データ品質スコアリング        ← どのレコードが信頼できるか
└─ 全量履歴の保持               ← S3に生データを全部残す（消さない）
```

## Synthetic Dataへの反映方針

リアリティを出すために、生成データに意図的に以下の「汚れ」を混ぜる。

- 同一人物が複数システムに別IDで存在する（名寄せ対象）
- 電話・住所・氏名のフォーマットばらつき
- 和暦・西暦混在
- 長期間購買のない休眠・チャーン顧客が多数含まれる
- 一部レコードに欠損値・異常値

## IT・データ部門の現状

| 項目 | 内容 |
|---|---|
| IT部門人数 | 5名 |
| IT部門の役割 | 社内インフラ・ヘルプデスク・基幹システム保守が中心 |
| データ専任 | ゼロ（兼任で少し触れる人が1名いる程度） |
| 現在のデータ管理 | ECはECシステムのDB、POSはPOS独自DB、連携なし |

### データのサイロ化（現状の課題）

```
ECサイトDB       → マーケ部門だけが見ている
POSシステムDB    → 店長・本部経営管理だけが見ている
会員アプリDB     → アプリ開発ベンダーが管理（社内に権限なし）
在庫管理DB       → 物流部門だけが見ている
```

同じ顧客がECでも店舗でも買っているのに、**同一人物として認識できていない**。

## 発注の動機・背景

> 「ヤマダ・ヨドバシには価格で勝てない。地域密着・接客で差別化してきたが、それだけでは限界。ECと店舗のデータをつなげて、マーケや店長が使えるようにしてほしい。社内に詳しい人間はいないので設計から構築まで任せたい。納品後は自社IT部門で運用できるようにしてほしい。」

### 優先したいこと

1. ECと実店舗の顧客データを統合して同一人物として管理したい
2. マーケ部門が施策の効果を自分たちで確認できるようにしたい
3. 「なんとなく」の勘ではなく、データに基づいた販促を打ちたい
4. 将来的にレコメンドや来店予測もやってみたい（優先度は低め）

### 制約・懸念

- 納品後のk8s・Kafka運用を自社IT部門（5名）が担う必要がある
- 大規模投資は難しい。段階的に価値を示しながら進めたい
- データのガバナンス・個人情報管理には敏感（顧客からの信頼が命）

## プロジェクトの構図

```
株式会社テクノマート（発注元）
    └─ IT部門 5名（運用・保守担当、データは素人）
    └─ マーケ部門（主なデータ利用者）
    └─ 店長・経営管理（KPIダッシュボード利用者）

        ↑ 受注・設計・構築・引き継ぎ

僕たち（外部データエンジニアリングベンダー）
    └─ データ基盤の設計・構築
    └─ 運用ドキュメント・引き継ぎ資料の整備
    └─ 納品後は保守サポート契約（月次）
```

## 本番 vs 今回の開発環境

| | 本番想定 | 今回のローカル環境 |
|---|---|---|
| クラウド | AWS | VirtualBox VM（Ubuntu LTS） |
| オーケストレーション | EKS | k8s on VM |
| ストレージ | S3, RDS | LocalStack |
| ストリーミング | Amazon MSK (Kafka) | Kafka on k8s |
| IaC | Terraform (AWS provider) | Terraform (LocalStack provider) |

> ローカルVM環境は本番AWS環境の縮小再現として位置づける。
> Terraformのコードは本番移行時に最小限の変更で対応できるよう設計する。

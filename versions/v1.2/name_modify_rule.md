# 名寄せルール定義

作成日: 2026-03-19
参照: verification/src/dirty.py, verification/src/customers.py

本ドキュメントは、EC / POS / アプリの3システムに散在する顧客レコードを
`unified_customers` に統合する際のクレンジング・マッチングルールを定義する。

---

## 1. 対象システムと汚れの出所

| システム | テーブル（raw） | 汚れの種類 |
|---|---|---|
| EC | `ec_customers` | 電話番号フォーマット / 都道府県表記 / メールアドレス腐敗 |
| POS | `pos_members` | 電話番号フォーマット / 生年月日が**和暦**（西暦なし） / 氏名が**全角スペース**区切り |
| アプリ | `app_users` | 電話番号フォーマット / 氏名がカジュアル（姓カナのみのケースあり） |

---

## 2. クレンジングルール

### 2-1. 電話番号の正規化

**目的**: 同一人物が3システムに異なるフォーマットで登録されているため、比較可能な形式に統一する。

**入力パターン（5種類）**:

| パターン | 例 | 出現確率 |
|---|---|---|
| 標準（ハイフンあり） | `090-1234-5678` | 40% |
| ハイフンなし | `09012345678` | 28% |
| +81形式 | `+81-90-1234-5678` | 10% |
| suffix付き | `090-1234-5678（携帯）` | 10% |
| 欠損（空文字） | `` | 12% |

**正規化手順**:
1. 末尾の `（携帯）` 等のsuffixを除去（`）` 以降を削除）
2. `+81` を `0` に置換（`+81-90-` → `090-`）
3. ハイフン・スペース・記号をすべて除去して数字のみにする
4. 11桁の数字であることを確認（070/080/090始まり）
5. 出力フォーマット: `XXX-XXXX-XXXX`（ハイフンあり統一）
6. 欠損（空文字・NULL・変換失敗）: `NULL` として扱う

```python
# 変換例
"090-1234-5678"       → "090-1234-5678"  # そのまま
"09012345678"         → "090-1234-5678"
"+81-90-1234-5678"    → "090-1234-5678"
"090-1234-5678（携帯）" → "090-1234-5678"
""                    → NULL
```

---

### 2-2. 生年月日の正規化（和暦→西暦変換）

**目的**: POSシステムは和暦文字列で保存しているため、比較のために西暦 `YYYY-MM-DD` に統一する。

**入力パターン**:

| 元号 | 記法 | 元年の西暦 | 変換例 |
|---|---|---|---|
| 昭和（S） | `S55` | 1926年 | `S55` → 1980 年（1925+55） |
| 平成（H） | `H15` | 1989年 | `H15` → 2003 年（1988+15） |
| 令和（R） | `R3` | 2019年 | `R3` → 2021 年（2018+3） |

> POSの `birth_date_jp` は年のみ（例: `S55`）。月・日はPOSテーブルに存在しない。

**正規化手順**:
1. 先頭文字で元号を判定（S / H / R）
2. 数字部分を抽出してオフセット加算
3. 出力: `YYYY-01-01`（月日不明のため1月1日で補完）
4. EC・アプリ側は `YYYY-MM-DD` で保存済みのためそのまま使用

```python
# 変換例
"S55"  → "1980-01-01"   # 1925 + 55
"H15"  → "2003-01-01"   # 1988 + 15
"R3"   → "2021-01-01"   # 2018 + 3
"1990-03-24"  → "1990-03-24"  # 西暦はそのまま
```

**注意**: 月日が補完値（01-01）の場合、氏名＋生年月日マッチングで**年のみ一致**を許容する緩やかなルールを適用する（→ セクション3参照）。

---

### 2-3. 都道府県コードの正規化

**目的**: EC・アプリで表記が3種類混在するため、正式名称に統一する。

**入力パターン（3種類）**:

| パターン | 例 | 出現確率 |
|---|---|---|
| 正式名称 | `東京都` / `神奈川県` | 60% |
| 略称 | `東京` / `神奈川` | 22% |
| JIS数字コード | `13` / `14` | 18% |

**対応テーブル（対象8都県）**:

| 正式名称 | 略称 | JISコード |
|---|---|---|
| 宮城県 | 宮城 | 4 |
| 福島県 | 福島 | 7 |
| 栃木県 | 栃木 | 9 |
| 茨城県 | 茨城 | 8 |
| 埼玉県 | 埼玉 | 11 |
| 千葉県 | 千葉 | 12 |
| 東京都 | 東京 | 13 |
| 神奈川県 | 神奈川 | 14 |

**正規化手順**:
1. 入力が正式名称テーブルに存在する → そのまま使用
2. 数字文字列 → JISコード逆引きで正式名称に変換
3. 略称 → 末尾に「県」または「都」を補完して正式名称に変換
4. いずれにも該当しない → `NULL`（データ品質ログに記録）

---

### 2-4. 氏名の正規化

**目的**: システム間でスペース種別・表記が異なるため、比較可能な形式に統一する。

| システム | 氏名のフォーマット |
|---|---|
| EC | 漢字フルネーム（半角スペース区切り） |
| POS | カナ（**全角スペース**区切り） |
| アプリ | 漢字フルネームまたは**姓カナのみ**（30%の確率） |

**正規化手順**:
1. 前後の空白・全角スペースをトリム
2. 全角スペース（`　`）を半角スペースに変換
3. 氏名マッチングはカナ読みでは行わない（アプリが姓カナのみのケースがあるため精度が低い）
4. マッチングには漢字氏名のみ使用。アプリの姓カナのみ表記はマッチングキーから除外

---

### 2-5. メールアドレスの扱い

**目的**: 古いアカウントはキャリアメール（バウンス）に腐敗しているケースがある。

**腐敗パターン**:
- アカウント年齢5年超のEC顧客の約25%が旧キャリアドメインに変化している
- 対象ドメイン: `docomo.ne.jp` / `ezweb.ne.jp` / `softbank.ne.jp` / `i.softbank.jp`

**扱い**:
- メールアドレスはマッチングキーとして使用するが、旧キャリアドメインの場合は**信頼度を下げて**使用（電話番号・氏名+生年月日で補完）
- 一致した場合でも confidence score を付与し、旧キャリアドメイン一致は低スコアとする

---

## 3. マッチングルール（名寄せロジック）

### 3-1. マッチングキーと優先順位

以下の順で評価し、最初に一致したキーでマッチとする。

| 優先度 | キー | 使用フィールド | 信頼度 |
|---|---|---|---|
| 1 | メールアドレス | `email`（正規化なし） | **高**（旧キャリアは中） |
| 2 | 電話番号 | `phone`（正規化後） | **高** |
| 3 | 氏名 + 生年月日 | `name_kanji` + `birth_date`（年のみ可） | **中**（月日補完時は低） |

### 3-2. 詳細ルール

**ルール1: メールアドレス一致**
- 完全一致（大文字小文字を無視）
- 旧キャリアドメイン（`docomo.ne.jp` 等）の場合: 一致でも単独マッチは避け、他キーとの AND 条件を推奨

**ルール2: 電話番号一致**
- 正規化後の `XXX-XXXX-XXXX` 形式で完全一致
- 欠損（NULL）同士は一致しない

**ルール3: 氏名 + 生年月日一致**
- 氏名: 漢字フルネームの完全一致（半角スペース正規化後）
- 生年月日:
  - 両方が完全な `YYYY-MM-DD` → 年月日すべて一致
  - 一方が `YYYY-01-01`（POSの和暦から補完）→ **年のみ一致**を許容
- どちらかの氏名がカナのみ（アプリ姓カナ）の場合: このルールは適用しない

### 3-3. 複数マッチの処理

- 複数のキーで同一レコードを指す場合: 最初に発見した `unified_id` に統合
- 矛盾するマッチ（キーAでは別人になる）: `resolution_score` を下げてフラグを立てる（手動確認候補）

---

## 4. customer_source_map への記録

マッチが確定したら `customer_source_map` に記録する。

```sql
INSERT INTO customer_source_map (unified_id, source, source_id)
VALUES (:unified_id, :source, :source_id)
ON CONFLICT (source, source_id) DO NOTHING;
```

| カラム | 値 |
|---|---|
| `unified_id` | `unified_customers.id`（UUID） |
| `source` | `'ec'` / `'pos'` / `'app'` |
| `source_id` | 各システムのID（`ec_user_id` / `member_id` / `uid`） |

---

## 5. unified_customers への統合レコード生成

| フィールド | 値の選択方針 |
|---|---|
| `canonical_name` | EC漢字氏名を優先。なければアプリ漢字氏名 |
| `email` | ECのメール（旧キャリア以外）を優先 |
| `phone` | 正規化後の電話番号（ECを優先、なければPOS/アプリ） |
| `birth_date` | EC西暦を優先。なければPOS和暦変換（`YYYY-01-01`） |
| `prefecture` | EC正規化値を優先 |
| `name_kana` | POSカナを優先（全角→半角スペース変換後） |
| `resolution_score` | マッチキーの信頼度に基づく 0.0〜1.0 |

## 6. 既存データとの整合性

現状（2026-03-19時点）:
- `unified_customers`: 5000件（環境構築時のテストデータ）
- `customer_source_map`: 8171件（ec:3462 / pos:2761 / app:1948）
- データ量は少ない。将来 `verification/generate.py --count` を増やして拡張予定

バッチ実行方針:
- **初回フルロード**: 全ステージングレコードを対象に名寄せを実行（既存の unified_customers を上書きしない。重複チェックして差分のみ追加）
- **日次差分**: `created_at > 前回実行時刻` のレコードのみ処理
